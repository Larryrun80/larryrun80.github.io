Field options / 字段可选项

null

是否可为空，默认False，如果需要在布尔型字段接受空值，请使用NullBooleanField。

blank

是否允许空值，与null的区别在于，null是纯粹数据库相关的，而blank是一个输入校验相关的。

choices

有两个元素组成的列表或元组集合，作为这个字段的可选输入项，前者是存储于数据库的真实值，后者是一个可读的命名。
通常建议在Model类中定义choices，并且用适当命名的常量来表征每个值。

这样，你可以在任何应用Student Model的地方使用 Student.SOPHOMORE 来表征’SO’。
对每一个choices set， Django提供了一个方法来获取可读命名： get_FOO_display()

针对choices， Django会自动在下拉框渲染一个’———‘行，如果你要覆盖这个行为，为choices增加一个None值： (None, ‘Your String For Display’), 当然你也可以使用 ’‘ 替代None。

db_column

对应的数据库的列名，如果不定义，Django会使用字段名。

db_index

对这个字段创建索引。

db_tablespace

表空间，不支持表空间的数据库类型将会被忽略。

default

默认值，可以是一个值或一个可调用的对象。如果是后者，每次新对象创建时，那个需要被调用的对象都会被调用。

editable

如果是False, 这个指端不会在admin后台或其他Form中显示，他们也将跳过类型检测，默认为True。

error_messages

如果定义会覆盖原生的错误方法，传入一个包含error key和你想变更的error message的dictionary。
error key： null，blank, invalid, invalid_choice, unique, and unique_for_date。 在 这里 查看更多。

help_text

在form渲染的时候会被展示，他也同样具有文档价值。 如果你需要展现出HTML的表现形式，可以定义HTML标签。

primary_key

定义主键， 如果你没有定义任何主键， Django会帮你自动生成一个AutoField类型的主键。
如果定义了primary_key=True， 同时也意味着定义了 null=False， unique=True。
一个对象只允许一个主键。
主键是只读的，如果你尝试变更他的值，Django会为你创建一个新的对象并保有旧的。

unique

如果是True，这个字段的值将在全表表示唯一。
这个选项在除了 ManyToManyField, OneToOneField, FileField 的字段可用。
如果一个字段的unique呗设置为True，你不需要再设置db_index。

unique_for_date / unique_for_month / unique_for_year

针对DateField或DateTimeField，保证他们对应的属性（日期、月、年）唯一。

verbose_name

字段的可读命名。如果不设置，Django会根据字段名自动创建。

validators

验证器，对输入进行验证，详见validators

Field types / 字段类型

AutoField

自增字段，整型

BigIntegerField

64位整数，取值范围 -9223372036854775808 到 9223372036854775807

BinaryField

存储二进制数据，只支持bytes类型的赋值。
不能应用queryset进行查询。

BooleanField

布尔型， 值为 true/false， 将被渲染成CheckboxInput。
如果你希望能采用null值，使用 NullBooleanField。
初始值为 None。

CharField

字符型，如果你需要存储大量文字，使用TextField。
CharField有一个必须指定的参数： max_length。

CommaSeparatedIntegerField

存储一组被逗号分隔的整数，max_length是必选参数。

DateField

对应Python中的datetime.date实例，有一些可选参数：

DateField.auto_now
每次对象保存时，自动更新当前时间，在’last-modified’这样的字段中非常有用。

DateField.auto_now_add
每个对象创建的时候，自动记录当前时间，

以上两个参数不能被重载。
这个字段会被默认渲染为文本框，在admin中已经添加了一个javascript日历控件，并且有一个’今日’快捷键。
这个类型也包含一个invalid_date错误信息key

DateTimeField

对应Python中的datetime.datetime类型，和dateField拥有相同的附加属性。

DecimalField

浮点数。两个必须参数：
DecimalField.max_digits : 数字的总长
DecimalField.decimal_places : 小数部分长度
前者必须大于等于后者。

EmailField

这个类型会自动做电子邮件验证，max_length默认值是75，目前75已经无法兼容RFC3696/5321标准的所有电子邮件，所以建议设置max_length为254，处于向下兼容的原因，默认值仍将保持为75.

FileField

文件上传字段。这个字段不支持primary_key和unique属性。

两个附加属性：
upload_to: 一个相对路径，相对于配置项中 MEDIA_ROOT 配置，也就是说，这个路径会挂在MEDIA_ROOT后面形成一个真实访问路径。
路径中可以包含strftime()格式，在文件上传时，这部分会被当时的日期/时间替代。例： photos/%Y/%m/%d
这个参数也可以是一个可调用的函数，这个函数必须接受两个参数并返回一个Unix风格的路径。
需要传入的参数是：

  • instance： 定义FileField的对象的实例，通常是正在被上传的文件所在的实例。需要注意的时，此时通常这个对象还没有被保存到数据库，如果使用了一个AutoField字段作为主键，此时这个对象应该还没有获得主键。
  • filename： 原文件名。

在Model中使用FileField和ImageField需要以下步骤：

  1. 在settings中定义MEDIA_ROOT和MEDIA_URL，并且确认web server用户对文件夹有写权限；
  2. 在Model中加入FileField或ImageField，定义upload_to参数定义上传到MEDIA_ROOT的某个子文件夹；
  3. 所有数据库中保存的都是一个相对于MEDIA_URL的路径，你可以使用Django提供的url属性反问道这个路径，例如，如果你的文件名是mug_shot，你可以在模板中这样获取绝对路径： ``

例： 如果你的MEDIA_ROOT定义为 /home/media，upload_to定义为photos/%Y/%m/%d，你在2007年1月15日上传了一个文件，这个文件将被存在文件夹：/home/media/photos/2007/01/15
请注意上传文件的安全问题，不要让人能够把一个CGI或PHP文件传到你的可访问目录。
更多信息请参考： Managing files

Model.

Django中的Model层最主要用来处理与数据库的交互，通过API在对象层级上方便地实现基于数据库的CRUD操作。

  • 通常来说，每一个Model映射到一个单一的数据库表。
  • 每个Model都必须继承 django.db.models.Model
  • Model的每一个属性都代表着数据库的一个字段
  • 基于此，Django提供了一个自动生成的数据库访问API

Using models

当你定义好了自己的model，你需要告诉Django你要使用它，方法是在项目的settings.py文件中，INSTALLED_APPS配置项中增加包含这个model的应用的引用。

如果你使用了manage.py startapp来创建这个应用，你可以直接写应用名。
做完这一步后，记得执行 manage.py migrate 命令，完成数据库层的更新

Field Types

每一个字段都应该是一个对应的Field类型，Django用这个类型确定：

  • 数据库中列的类型（如，INTEGER， VARCHAR）
  • HTML部件会依据这个类型决定如何渲染这个字段（文本框或下拉框等）
  • 实现输入数据验证
    如果你觉得当前的字段类型无法满足你的需要，也可以自定义字段类型。

Field Options

每个字段类定义时可以跟一些对应的参数，例如，CharField需要一个mas_length参数设置允许的文字最大长度。下面例举一些常见的参数：

  • null
    是否允许为空，默认False
  • blank
    是否允许空白，默认为False。（注意这是Django的一个数据验证，和null不一样，null是指是否可以不设置任何值）
  • choices
    允许的取值的列表，会在html页面被渲染成选择框
    example：

每个元组都包含两个成员，第一个是用来存储在数据库中的值（如 FR）， 第二个是被现实在HTML页面中的值。你可以用get_FOO_display方法查看这个值，例如：

  • default
    字段的初始值
  • help_text
    在form中，现实在help信息里面的文字，如果不使用form，这个设置对于文档化也有一定好处
    -primary_key
    主键，如果被设置为True，这个字段会被看做model的主键。
    每个表必须有一个主键，如果你没有设置任何字段为主键，Django就会自动帮你创建一个自增的id，如果你设置了主键，这个id列就不会被自动创建
    主键是只读的，如果你尝试修改它，会创建一个新的对象。
  • unique
    如果是True，Django会帮你确认这个字段的值在全表必须唯一。

枪炮、病菌与钢铁，人类社会的命运-reading

随笔 . Chapter 2

在波利尼西亚人这个来自自然的例子里，可以看到影响人类社会发展的因素是多样的，错综复杂的，叠加的。仅在波利尼西亚，我们就能够看到岛屿气候、地质类型、海洋资源、面积、地形的破碎程度等因素叠加的作用，使得处于同一起跑线的人们在短短一千年间就产生了巨大的差异。
这是一个有趣的观点：人类选择生存和发展的地域在开始看来是具有极大偶然性的——他们走到了某个地方，觉得这里可以生存，就定居下来。然而这个聚居地的地理、气候等因素，又必然性地决定了这个群体未来的命运——例如，能够从事集约型生产（气候合适，利于灌溉）的，能够承载密集人口的，资源丰富的区域，能够首先完成粮食的剩余、释放生产力用以专业化（领袖、士兵、宗教、手艺人、官员等），进一步优化资源配置。他们必然进一步形成了阶级，建立帝国的雏形——一切都是如此必然。

看一个反面的例子，莫里奥里人：

  • 他们居住在亚南极地区，气候寒冷，祖先的热带作物无法生长，他们只能回到狩猎采集生活，无法贮藏农作物，所有人每天只能为填饱肚子奔波
  • 他们主要捕食的海鸟、鱼类等，不需要复杂的技术，导致技术无法发展
  • 他们居住的都是较小、偏远的岛屿，难以与外界沟通、贸易，也无法承载更多的人口，总人口维持在2000左右他们甚至采用阉割一些男婴来减少人口过剩的潜在冲突
  • 由于没有其他可以到达的岛屿，他们只能学会彼此和睦相处，宣布放弃战争，形成一个不好战的群体，技术武器简陋，没有强有力的领导和组织

这样的群体，面对外来威胁的时候，后果是可想而知的。
要发展，第一件事是解放生产力，人类发展史无不如此。

农民的力量 . Chapter 4

集约生产，相对于狩猎采集而言，对早期人类的影响是巨大的：

  • 野生动植物中只有一小部分能供人类食用，这样，饲养和种植使人们能够从每英亩土地获得百倍的卡路里，这为人口的发展提供了一个主要前提；
  • 解放人口增长的第二个制约：定居。这可以使得女性能够连续剩余，而狩猎为生的部落由于需要不停移动，在一个孩子不成为累赘前很难再次生育；
  • 驯化动物和种植植物的相互作用：动物粪便成为植物的肥料，动物可以帮助拉犁，帮助人们开拓更多地土地，动物的粪便是早期燃料的主要来源；
  • 定居使得食物可以贮藏，这样就能够养活非生产的专业人员，这些专业人员首先是国王和官员，通过税收和国家建制形成一整套粮食生产、贮藏、分配的体系。这种统一的行政体系和阶级极大地促进了科技和语言的发展。其次的专业人员包括用以征服的职业军人，为战争和社群提供宗教服务的神职人员，铸造刀剑枪炮的工艺者，能够保存信息的抄写员等；
  • 动植物提供了大量的天然纤维，用来制作服装、毯子、网、绳子以及其他工具、制品；
  • 大型哺乳动物成为主要的运输手段，以及马在战争中的统治性地位，这种地位知道一站中卡车和坦克诞生才被取代；
  • 驯养动物的过程中产生了针对人的专业化病菌，随之产生的时针对这些病菌的免疫力，当有免疫力的人面对没有免疫力的人时，病菌在征服的过程中也扮演了重要的角色。
